1. Service Broker — the platform front door

~03:30–09:30 in the talk. The first system the engineer built.

Atlassian runs internally on a Kubernetes-style declarative resource model. Internal developers don't call AWS directly; they ask the platform for resources (databases, load balancers, edge routes) and the platform handles the provisioning. The API surface for these requests is an Open Service Broker implementation.

02:14It's sort of built to operate in a Kubernetes world where you're submitting these provisioning requests as things… You might provision something like a database and then you'll get MySQL. Something that's SQL-compatible but abstracted away for your internal developers.

2. Sovereign — the Envoy control plane

~09:30–14:30. The custom xDS server, open-sourced on Bitbucket.

The platform's mandate was to replace every team's individual AWS load balancer with a single, centrally managed, fully programmable edge. The engineer chose Envoy as the data plane (industry-standard, dynamic xDS config API) and built a custom control plane on top, named Sovereign.

11:49I open-sourced this software and I called it Sovereign. You can actually go find that on Bitbucket. It's a public repo at least for now.

Sovereign renders Envoy resources (clusters, routes, listeners) from templates + live context (broker DB state + S3 data). Envoy proxies fetch their config via xDS — pull model, no restarts when config changes.

4. AWS deployment topology

~14:30–21:00. CloudFormation + pre-baked AMIs + Auto Scaling. Immutable infrastructure.

The Envoy fleet runs on EC2, provisioned by CloudFormation, with all software pre-baked into the AMI. When config changes, you don't ssh in — you bake a new AMI and rotate via the Auto Scaling Group.

5. Programmable proxy with sidecars

~26:00–31:00. Auth (in Rust), authorization, and rate limiting bolt onto every Envoy.

Once the centralized edge owned all ingress, it became the natural place for cross-cutting concerns. The team added a sidecar layer that runs locally next to every Envoy and handles concerns that previously lived in each microservice.

30:01The authentication sidecar was created by me, written, of course, in the Lord's language, Rust. Authorization was done by another team and rate limiting was done by another team.

The sidecars receive dynamic config over the wire, just like Envoy itself — making the whole edge a multi-tenanted programmable platform.

6. Migration — the multi-year platform-engineering effort

~21:30–25:30. Every Atlassian product team had to be moved off its own load balancer onto the centralized edge.

Building the platform was the easy half. The other half was convincing every product team to migrate.

7. Tech stack summary

LayerTechnologyWhy
Broker API / Control plane FastAPI Python Fast iteration, native async, good Pydantic models for Envoy resource schemas.
Async queue AWS SQS Durable, decouples API from worker provisioning time (which can be minutes).
State store AWS DynamoDB Multi-region, single-digit-ms reads, schema-flexible for evolving resource types.
Config / data side store AWS S3 Side context for Sovereign rendering (configs the broker doesn't own).
Data-plane proxy Envoy Dynamic xDS config, programmable filters, the industry-standard service-mesh data plane.
Edge auth sidecar Rust Performance + memory safety at the edge. Speaker calls Rust “the Lord's language.”
Compute AWS EC2 + ASG Auto-scaling, immutable AMI-based rotation, region isolation.
IaC CloudFormation + AMI One CFN template per region; rebake AMIs on config change. Immutable infra.
Edge networking Route 53 + ACM + CloudFront + NLB DNS, TLS certs, CDN, L4 load balancer in front of the Envoy fleet.
Open-source control plane Sovereign (Bitbucket) Speaker's own open-source contribution. Custom xDS server on FastAPI.
Provisioning spec Open Service Broker API Kubernetes-ecosystem standard for declarative resource provisioning.

8. Why this is interesting in 2026 hindsight

  • Textbook control-plane / data-plane split. Sovereign as xDS server, Envoy fleet as data plane — the same pattern that Istio, Linkerd, and AWS App Mesh use. Atlassian built theirs in-house when much of the off-the-shelf tooling didn't exist or wasn't production-ready yet.
  • Open-sourcing the control plane. Sovereign on Bitbucket is a strong cultural signal — Atlassian let platform engineers open-source infra they built on the clock. (Worth checking if it's still maintained; the speaker hedges with “public repo at least for now.”)
  • Rust at the edge in 2018–2020. The auth sidecar choice was forward-looking; most edge auth at the time was Go (Envoy itself is C++). Rust adoption for sidecars only became mainstream around 2022–2023.
  • The migration is the real moat. Many orgs build a shiny centralized edge that nobody uses. Atlassian moved Jira, Confluence, Statuspage, and the rest of the portfolio onto it. That's the unglamorous multi-year work that actually matters.
